# מדריך אבטחה והתקנה — منصّة البطولات

מדריך זה מסביר **בדיוק** מה צריך לעשות אחרי העלאת הקוד המעודכן, לפי הסדר. חלק מהצעדים חובה (אחרת ההרשאות יישברו), וחלק מומלצים.

> ⚠️ **סדר הפריסה קריטי.** הרשאות הכתיבה החדשות דורשות מייל מאומת. הבעלים (owner) חייב לאמת את המייל שלו **לפני** פרסום ה-rules החדשים, אחרת גם הוא יינעל החוצה.

---

## שלב 0 — סדר הפעולות (חובה, לפי הסדר הזה)

1. **העלה את הקוד המעודכן** ל-GitHub (Upload files) ופרסם. עדיין **אל תפרסם את ה-rules**.
2. **היכנס לאפליקציה עם חשבון הבעלים** (`ahmad.kefah11sar@gmail.com`).
   - אם זו התחברות רגילה (מייל/סיסמה): המסך יבקש **לאמת את המייל**. לחץ על הכפתור לשליחת מייל אימות, פתח את הרסלת האימות שהגיעה למייל, לחץ על הקישור, וחזור לאפליקציה ולחץ **«تحقّقت — تحديث»**.
   - טיפ: אם המייל לא מגיע, בדוק בתיקיית ספאם.
3. רק אחרי שהבעלים מאומת בהצלחה ורואה את לוח הניהול — **פרסם את ה-rules החדשים**:
   `Firebase Console → Firestore Database → Rules → הדבק את תוכן firestore.rules → Publish`.
4. בדוק שהכול עובד: צור/ערוך טורניר, הוסף מדינה/קבוצה/משחק, הזן תוצאה.

> אם דילגת וכבר פרסמת את ה-rules לפני אימות הבעלים — לא נורא: פשוט אמת את מייל הבעלים כרגיל (שלב 2). האימות עובד ברמת Firebase Auth ואינו תלוי ב-rules.

**חשבונות קיימים:** כל חשבון שכבר נרשם בעבר (כולל הבעלים ומנהלים) יצטרך **פעם אחת** לאמת את המייל שלו כדי לחזור לפעול. משתמשי Google מאומתים אוטומטית ולא צריכים לעשות כלום.

## שלב 1.5 — חשבונות משתתפים למסבקת התחזיות «مسابقة التوقّعات»

מסבקת התחזיות דורשת חשבון פלטפורמה אמיתי עם username ייחודי. אם המשתמש מזין מייל ומאמת אותו — החשבון פעיל מיד. אם הוא נרשם בלי מייל — הוא יכול להתחבר עם username + סיסמה, אבל נשאר בהמתנה עד שהבעלים מאשר אותו. אין יותר הרשמה בשם זמני ואין צורך ב-Anonymous Auth או Phone Auth בשביל המשתתפים.

`Firebase Console → Authentication → Sign-in method`:

1. ודא ש-**Email/Password** פעיל. גם חשבונות בלי מייל משתמשים בזהות פנימית כדי לאפשר התחברות חוזרת עם username + סיסמה.
2. מומלץ להפעיל גם **Google** כדי לאפשר כניסה מהירה.
3. אין צורך להפעיל **Anonymous**. אם הוא פעיל מפריסה ישנה, החוקים החדשים עדיין לא יאפשרו לו להירשם או לשמור תחזיות.

> ה-rules החדשים (`firestore.rules`) כבר כוללים את האוספים: `usernames` (נעילת username ייחודי), `users` (כולל מצב approved), `pcomps` (מסבקות), `predictors` (שם חשבון המשתתף — קריא לכולם לצורך טבלת הדירוג), `predictorContacts` (טלפון/מייל/גיל — **פרטי**, קריא רק לבעל הרשומה ולמנהל הטורניר), ו-`predictions` (התחזיות). ודא שפרסמת אותם (שלב 0.3).

### פרטיות המשתתפים
- **שם החשבון** גלוי בטבלת הדירוג הציבורית.
- **טלפון/מייל/גיל** נשמרים באוסף נפרד ומוגנים ב-rules — רק המשתתף עצמו ומנהל הטורניר יכולים לקרוא אותם (דרך «👥 المشاركون» בלשונית «التوقّعات» בניהול).
- **הוגנות:** התחזיות ננעלות אוטומטית כשהמשחק עובר לסטטוס «مباشر/منتهية». התחזיות עצמן קריאות ציבורית (נדרש לחישוב טבלת הדירוג בצד-לקוח); בממשק מוצגות תחזיות של אחרים רק אחרי הנעילה.

### כדי שמיילי אימות לא ייפלו לספאם
1. `Authentication → Templates` → בחר שפה (עברית/ערבית) ושנה את **Sender name** לשם ידידותי (למשל «منصّة البطولات»).
2. הדומיין `ahmad-sarsor.github.io` חייב להיות ב-Authorized domains (שלב 3).
3. כתובת השולח היא `noreply@<project>.firebaseapp.com`; אם צריך אמינות גבוהה יותר, השתמש בדומיין מותאם אישית דרך Templates → Customize domain.

הקוד כבר בנוי סביב זה: הודעות השגיאה בהתחברות גנריות («الم/كلمة المرور غير صحيحة»), ו«نسيت كلمة المرور» תמיד מציג הודעת הצלחה גם אם המייל לא קיים.

## שלב 4 — הקשחת מפתח ה-API (מומלץ מאוד)

מפתח ה-API של Firebase גלוי בקוד הצד-לקוח (זה תקין ומתוכנן), אבל כדאי להגביל אותו כדי שלא ישמש מאתרים אחרים.

`Google Cloud Console → APIs & Services → Credentials`:
1. פתח את מפתח ה-API של הפרויקט (Browser key).
2. תחת **Application restrictions** בחר **HTTP referrers (web sites)**.
3. הוסף את ה-referrers:
   - `https://ahmad-sarsor.github.io/*`
   - `http://localhost/*` (לפיתוח)
4. שמור.

> שים לב: אחרי ההגבלה, כל דומיין אחר שינסה להשתמש במפתח ייחסם. אל תגביל בטעות דומיינים שאתה באמת משתמש בהם.

## שלב 5 — App Check עם reCAPTCHA v3 (מומלץ — ההגנה מפני ספאם וניצול מכסה)

App Check מוודא שרק האתר שלך (ולא סקריפטים זדוניים) ניגש ל-Firestore. בלעדיו, סקריפט חיצוני יכול למלא את המסד בהצעות או נתוני ספאם עד מיצוי המכסה החינמית היומית. **קוד האתחול כבר מוטמע** ב-`assets/js/firebase.js` — נשאר רק:

1. `Firebase Console → App Check → Apps` → רשום את אפליקציית הווב עם **reCAPTCHA v3** (זה יוצר Site Key).
2. הדבק את ה-Site Key ב-`assets/js/config.js`:
   ```js
   export const APP_CHECK_SITE_KEY = "6L…";   // המפתח מהקונסול
   ```
   פרסם ל-GitHub Pages. מרגע זה הדפדפנים מתחילים לשלוח אסימוני App Check (עדיין בלי אכיפה — שום דבר לא נחסם).
3. המתן יום-יומיים, בדוק ב-`App Check → Metrics` שרוב הבקשות **Verified**, ורק אז הפעל **Enforcement** ל-Firestore. (אכיפה מוקדמת מדי עלולה לחסום גולשים לגיטימיים עם תוספי חסימה.)

## מודל ההרשאות — סיכום

| תפקיד | היכן מוגדר | מה מותר |
|---|---|---|
| **בעלים (owner)** | קבוע בקוד (`config.js` + `firestore.rules`) | הכל: מאשר מנהלי פלטפורמה וחברים, מנהל את כל הטורנירים |
| **מנהל פלטפורמה** | `admins/{email}` — מוענק ע"י הבעלים | מנהל את **כל** הטורנירים |
| **חבר מאושר (member)** | `members/{email}` — מוענק ע"י הבעלים | יוצר טורנירים ומנהל את שלו |
| **מנהל טורניר** | `admin_emails` על מסמך הטורניר | מנהל את אותו טורניר (למעט הוספת צוות ומחיקת הטורניר) |
| **מזין תוצאות (scorer)** | `scorer_emails` על מסמך הטורניר | **רק** הזנת תוצאות/סטטוס משחקים, אירועים (גולים/כרטיסים), והוספת שחקנים חדשים |

**מזין תוצאות** הוא התפקיד החדש: אידיאלי למישהו שיושב במגרש ומעדכן תוצאות בזמן אמת, בלי לתת לו גישה לערוך את מבנה הטורניר, הקבוצות או ההגדרות. מוסיפים אותו בכרטיס «طاقم البطولة» בלשונית «التفاصيل» של הטורניר.

### למה עכשיו בטוח להוסיף מייל מראש?

בעבר הייתה בעיה: אם היית מוסיף מייל להרשאה כלשהי, **כל אחד** שנרשם עם אותו מייל היה יורש את ההרשאה — גם אם המייל לא באמת שלו. עכשיו זה נסגר: **אף הרשאת כתיבה לא ניתנת אלא למייל מאומת** (`email_verified == true`). מי שנרשם עם מייל שאינו שלו לא יוכל לאמת אותו, ולכן לא יקבל שום הרשאה. לכן בטוח להוסיף מראש מיילים של מנהלי טורניר ומזיני תוצאות עוד לפני שהם נרשמו — ההרשאה תיכנס לתוקף רק כשייצרו חשבון ויאמתו את המייל.

בכרטיס הצוות תראה חיווי לצד כל מייל: «لم يُسجَّل بعد» (עדיין לא נרשם), «سجَّل لكنه لم يؤكّد بريده» (נרשם אך לא אימת), או השם שלו אם הוא מאומת.
